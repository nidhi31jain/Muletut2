A mule application can have one or more flow. The application accepts a message through an inbound endpoint in a flow and processes the message through a series of processors.
Typically, a flow has
  • A message source for accepting a message.
  • Message Processor(s) to process the message.
  • Can have an exception strategy also for better error handling.
A flow can either process the message by itself or route the message to other flows or sub flows to perform specific tasks. Simply, a flow is a chain of message processors.

Flows without message source are called private flows.

Sub Flow

  • Promotes code reuse, avoid code redundancy
  • It processes messages synchronously, i.e; in the same thread.
  • Inherits the exception strategy and processing strategy of the calling flow.

A single flow can reference to several sub flows and a sub flow can be referenced by several flows.

Larger flows should be broken into multiple flows and sub flows if possible, it makes for a cleaner view.
  • Makes the graphical view easy to use and understand.
  • XML code is easier to read.
  • Easier to test.
  • Code reuse.
